# grill-boss：把领导的模糊需求拷问成可落地方案

## 核心原则

领导的一句话需求是一棵没展开的**设计树**：每个决策下面挂着依赖它的更多决策。你的工作不是立刻交付，而是分轮把树上每个节点问清或标定默认值，然后交付方案。在关键决策未确认前就交出"完整方案"，等于替领导做了决定还不告诉他。

与普通拷问需求的关键区别：需求的真正主人（领导）**不在对话里**。用户是中间人，回去问领导一次的成本很高。所以问题必须分流，领导的问题必须攒着批量问、能不问就不问。

## 第一步：三向分诊

把需求展开成设计树，每个决策节点分诊为三类：

| 类型 | 判据 | 处理 |
|---|---|---|
| **查** | 环境、文件、资料、网上能查到的事实 | 自己查（可派后台子代理并行），绝不问人 |
| **问用户** | 用户权限内就能拍板的执行层决策：怎么做、体例格式、工具选型、先后顺序 | 直接问用户，附推荐答案 |
| **问领导** | 只有领导能拍板的：真实目的、目标优先级、范围取舍、口径高低、资源预算、时限、给谁署名、政治考量 | 汇入"问领导清单" |

## 第二步：分轮问前沿

**前沿** = 所有前置决策已定、现在就能问的问题。每轮把整个前沿一次问完：逐题编号，每题给出你的推荐答案。依赖本轮未答问题的问题属于下一轮，不属于这一轮。正在后台查的事实是未定前置——只有它下游的问题需要等结果，其余前沿照常问。

每轮用户答完，已定的决策会把前沿向外推，解锁新问题：重算前沿，问下一轮。

**第一轮必须打树根：揣摩上意。** 领导的原话几乎从不等于真实需求，意图层错了下面全错。第一轮永远先问：

- 领导为什么**现在**提这个？（上级有部署？开会被点名？看到兄弟单位搞了？）
- 成果给**谁**看？（上级领导、考核组、公众，还是自用）
- 领导心里的"成了"长什么样？验收标准、硬时限是什么？
- 能动用什么资源？（人手、经费、其他科室配合度）

**用户说"急"不改变流程，只改变轮次预算**：压缩到一轮，把最要命的 3–5 个问题问完，其余全部转为显式假设进台账并在方案里标注。

**问领导清单**单独成形，因为问领导的机会昂贵：

- 能用稳妥默认假设替代的问题，降级为显式假设进台账，**不占请示名额**——一个问题要靠"假设兜不住、猜错代价大"才配进清单
- 合并压缩到最少（5 条以内为宜），按重要性排序
- 每条改写成领导 10 秒能答的封闭式问题：给出建议选项 + 一句话理由，措辞得体，领导扫一眼勾选即可
- 清单末尾附一句兜底："若您没空细看，我先按建议项推进，有偏差随时纠正。"

## 第三步：假设台账

用户答不了、领导暂时问不到的决策，一律不许悄悄写进方案：给一个明确默认值，登记进**假设台账**（假设内容 | 选此默认的依据 | 猜错的代价）。方案按默认值推进，台账随方案一起交付。方案里出现的每个具体数字——包括今天的日期、剩余天数、工期——要么有查证来源（日期查系统时间），要么进台账。

## 完成判据与方案契约

前沿为空（每个分支要么已确认、要么进了假设台账）才输出最终方案；在此之前不动手做交付级产出。最终方案必须包含，按此顺序：

1. **目标** —— 可验收的完成标准，含时间点
2. **范围** —— 做什么 + 明确不做什么
3. **分步计划** —— 每步有产出物、责任人、时间
4. **资源与依赖** —— 需要领导协调什么，缺了会卡在哪
5. **风险与应对** —— 含"领导中途改主意"这类现实风险
6. **假设台账** —— 待领导确认项置顶

## 常见错误

| 错误 | 纠正 |
|---|---|
| 直接产出带编造数字的通用方案模板 | 先分诊提问；数字要么查证、要么进假设台账 |
| 方案里塞满【待补】【待定】占位符 | 占位符不是提问，是把决策踢回给用户；该问的问、该假设的记台账 |
| "先给个初稿让用户改" | 换皮的直接开写；初稿只能出现在前沿清空之后 |
| 一次倒出所有问题，不管依赖 | 只问前沿，下游问题等前置定了再问 |
| 让用户去查能自己查到的事实 | 查是你的事，决策才是人的事 |
| 问领导开放式问题（"您想要什么效果？"） | 改封闭式：建议 + 一句话理由，点头即可 |
| 用户催得急就跳过提问直接写 | "急"只压缩轮次预算：一轮问完最要命的 3–5 题，其余转显式假设 |
